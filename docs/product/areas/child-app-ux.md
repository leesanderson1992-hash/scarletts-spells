# Child App UX

## Purpose

This document defines the child-facing UX truth for learning surfaces.

It focuses on:
- what the child should see
- what should stay hidden
- how progress should feel

## Child experience principles

ADLE-specific child experience principles are governed by
[docs/product/areas/adle-child-experience-principles.md](/Users/katiesanderson/Documents/Scarletts%20Spells/scarletts-spells/docs/product/areas/adle-child-experience-principles.md:1).

The child app should feel:
- calm
- directed
- readable
- rewarding without being noisy

The child should not need to reason about:
- parent planning structures unless they are directly meaningful
- admin-only validation rules
- compatibility storage details

## Progress course child truth

The child should experience:
- grouped phases
- clear next module
- ordered progression
- understandable lock/unlock flow

The child should not need to manage the phase/module planning system directly.

## Timed course child truth

The child should experience:
- current rhythm
- recurring work
- support tasks
- checkpoints/review rhythm

The child should not experience the legacy timed module-first planning model as the main frame.

## Progress and pacing truth

Child-visible progress should come from shared selectors/contracts.

That includes:
- recurring monthly totals
- all-time recurring totals
- recurring pacing
- course goal pacing where surfaced

Do not create child-visible progress truth with page-local calculations.

## Warning and missed-event rule

Missed-event warnings are not a child-facing punitive model in v1.

Weekly missed-event truth exists for parent insight/review surfaces.
Child views should not duplicate missed backlog cards because of that parent warning model.
