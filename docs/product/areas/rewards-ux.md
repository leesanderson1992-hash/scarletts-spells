# Rewards UX

## Purpose

This document defines the product-facing UX truth for rewards.

It exists to keep:
- reward language
- child feedback
- parent expectations
- reward visibility

aligned with the canonical reward contracts.

## Core UX rule

Rewards should feel:
- motivating
- predictable
- non-exploitable

The UI should not imply reward rules that the system does not actually enforce.

## Product-level reward split

Current direction:
- Gold Coins reward course-related work
- spelling-specific reward states remain separate

The app should not blur:
- course progress rewards
- spelling mastery rewards

## Recurring task reward truth

Recurring tasks should not imply approval-based behavior unless the canonical reward logic supports it.

UX should clearly distinguish:
- progress only
- reward on completion
- reward on approval
- reward at target

The visible labels must match the actual reward-trigger rules.

## Parent vs child reward presentation

Parent UX should show:
- configuration
- payout expectations
- rule clarity

Child UX should show:
- earned outcome
- progress relevance
- celebration without rule confusion
